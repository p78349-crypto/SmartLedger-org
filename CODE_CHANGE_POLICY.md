CODE 변경 정책 — 지출 입력(Expense Input) 모듈

작성일: 2026-01-01
작성자: 작업자(개발자)

요약
- `지출 입력` 기능(특히 `lib/screens/transaction_add_screen.dart` 및 해당 파일이 의존하는 최근입력 저장/불러오기 로직)은 완성되어 QA를 통과했습니다.
- 이 모듈은 향후 변경 시 앱 동작에 즉시 영향을 주므로 기본적으로 "수정 금지"로 지정합니다.

적용 범위
- 파일: `lib/screens/transaction_add_screen.dart`
- 관련 로직: 최근입력 저장/불러오기(SharedPreferences 연동), 입력 유효성 검사, 카테고리 매핑

예외 및 변경 절차
1. 코드를 수정해야 하는 긴급한 이유가 있는 경우(보안·치명적 버그 등), 변경 제안서는 다음 정보를 포함해야 합니다:
   - 변경 목적 및 영향 범위
   - 테스트 계획(수정 후 수행할 테스트 목록)
   - 변경 승인자(프로젝트 소유자 또는 팀 리드 서면 승인 필요)
2. 승인 없이 직접 변경하는 것은 금지합니다.
3. 승인 후 변경은 별도의 브랜치에서 작업하고 PR(풀 리퀘스트)을 통해 코드 리뷰와 QA를 거쳐 메인 브랜치에 병합합니다.

연락처
- 변경 승인 및 질문: 프로젝트 소유자(또는 저장소 관리자)에게 직접 문의하십시오.

비고
- 이 문서는 저장소의 정책이며, 필요 시 유지보수 일정에 맞춰 업데이트합니다.

앱 페이지 정체성 (1–15페이지)
- 적용일: 2026-01-02
- 목적: 앱의 네비게이션 및 기능 정체성을 고정하여 자동화 흐름과 연결성을 보존합니다. 페이지 번호와 명칭은 다음과 같습니다.
   1. 대시보드
   2. 요리,쇼핑,지출
   3. 수입 (자산 보안 적용)
   4. 통계
   5. 자산
   6. ROOT
   7. settings menu
   8-15. 미래 확장(예약됨)

- 변경 규칙:
   - 페이지 정체성(번호/명칭) 변경은 반드시 문서화된 승인 절차를 거칩니다.
   - 정체성 변경 제안 시 영향 분석(데이터 흐름, 자동화 연결), 테스트/QA 계획, 승인자를 포함한 변경 제안서를 제출해야 합니다.
   - 승인 전에는 코드나 UI에서 임의로 페이지 번호/기능을 재배치하면 안 됩니다.

- 문의: 변경 승인 및 정책 관련 질문은 프로젝트 소유자에게 문의하십시오.

완료된 기능 코드 변경 금지
- 적용일: 2026-01-02
- 목적: QA를 완료하고 운영 중인 기능(완성된 기능)은 예기치 않은 회귀와 사용자 경험 손상을 막기 위해 코드 변경을 제한합니다.
- 적용 범위: 저장소 내에서 QA를 통과한 모든 기능 모듈(예: `지출 입력` 모듈) 및 이와 직접 연관된 보조 로직.

- 규칙:
   - "완료된 기능"은 QA 레이블(또는 문서화된 완료 기록)을 가진 기능으로 정의합니다.
   - 완료된 기능의 소스 파일을 변경하려면 다음을 반드시 충족해야 합니다:
      1. 변경 제안서(목적, 영향 범위, 롤백 계획) 제출
      2. 승인자(프로젝트 소유자 또는 팀 리드)의 서면 승인 확보
      3. 자동화/수동 테스트 계획(변경 후 실행할 테스트 목록) 포함
      4. 별도 브랜치에서 개발하고 PR로 리뷰 및 QA를 통과할 것
   - 긴급 보안 패치나 치명적 결함은 예외로 하며, 변경 후 즉시 담당자에게 보고하고 사후 감사 로그를 남길 것.

   - 사용자 지시에 따른 변경 허용:
      - 제품 소유자 또는 지정된 사용자의 명시적 지시(이메일, 이슈, 티켓 등 문서화된 형태)가 있는 경우에 한해 변경을 허용합니다.
      - 해당 지시 문서는 변경 PR에 첨부되어야 하며, 승인자(프로젝트 소유자 또는 팀 리드)의 확인을 반드시 받아야 합니다.

- 위반 시 처리:
   - 허가 없이 완료된 기능을 변경할 경우 변경을 되돌리고 관련 커밋/PR에 대한 조사 및 기록을 진행합니다.
   - 반복적인 무단 변경은 권한 제한 조치로 이어질 수 있습니다.

- 문의 및 승인 요청: 정책 상의 예외나 변경 필요성은 프로젝트 소유자에게 서면으로 요청하십시오.
