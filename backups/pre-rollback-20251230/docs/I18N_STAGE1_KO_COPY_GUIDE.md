# I18N Stage 1 — Korean Source Copy Guide

목표: 한국어 원문을 **번역(EN/JA)하기 쉬운 형태**로 정리한다.
- 문장 구조가 단순하고 짧다.
- 용어가 화면 전체에서 일관된다.
- 직역해도 의미가 유지된다.
- 날짜/통화/단위는 문자열에 박지 않고 로케일 포맷터로 처리한다.

## 1) 원칙

### A. 짧고 중립적인 문장
- 권장: 명사형/간단한 서술형
  - 예) “저장했습니다.”, “삭제했습니다.”, “거래가 없습니다.”
- 지양: 과한 존댓말/구어체/군더더기
  - 예) “저장해놓았습니다”, “~해드렸습니다”, “~하시면 됩니다”

### B. 용어(명사) 먼저 고정
동일한 기능/데이터는 항상 동일한 단어를 사용한다.

### C. (한국) 영어 표현 사용 허용
대한민국 사용자 환경에서는 영어 UI 용어를 그대로 사용해도 자연스럽다.
다만 **혼용이 늘면 번역/검수 비용이 커지므로**, 아래 중 하나로 규칙을 고정한다.
- 규칙 1: 한국어 우선(필요 시 괄호로 영어 병기)
  - 예) “설정(Settings)”, “통계(Stats)”, “리포트(Report)”
- 규칙 2: 영어 우선(한국어는 최소화)
  - 예) “Settings”, “Stats”, “Report”, “Theme”, “Language”

한 화면/한 기능에서는 규칙을 섞지 않는다.

### (한국) 영어 단독 표기 후보
아래는 한국에서도 영어로만 표기해도 비교적 어색함이 적은 후보들이다.
- Settings, Theme, Language, Report, Stats, Category

※ 최종 결정은 “규칙 1(한국어+영어 병기)” 또는 “규칙 2(영어 우선)” 중 하나로 고정 권장.

## 3) 문자열 작성 패턴

### A. 변수 포함 문장
- 권장: 문장을 쪼개기
  - 예) “총액” / “{amount}” 처럼 라벨과 값 분리
- 지양: 조사(을/를/은/는)에 변수 끼우기

### B. 날짜/기간
- 원문에 날짜 포맷을 박지 않는다.
- UI 표시에서만 로케일 기반 포맷터를 사용한다.

## 4) 체크리스트
- 같은 개념이 다른 단어로 섞이지 않는가?
- 문장이 길어서 버튼/타이틀 오버플로우 위험이 없는가?
- 직역 시 의미가 달라지지 않는가?
- 한국어 조사/어순 의존이 강하지 않은가?
